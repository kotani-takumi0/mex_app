# MEX App 客観評価プロンプト

以下のプロンプトをそのまま外部AI（ChatGPT, Gemini, Claude等）に貼り付けて使ってください。

---

## プロンプト（ここから貼り付け）

```
あなたはプロダクトマネージャーとテックリードの両方の視点を持つシニアアドバイザーです。
以下のWebアプリケーションの概要・設計思想を読み、**ビジネス面・技術面・UX面**の3軸でバランスよく客観的に評価してください。

忖度や過度な褒めは不要です。改善すべき点は率直に指摘してください。

# プロダクト概要

## プロダクト名
**MEX（My Engineering Experience）**

## 解決したい課題
AI支援開発（いわゆる「バイブコーディング」）の普及により、誰でもアプリを作れる時代になりました。
しかし企業側から見ると、完成物だけでは「AIが作っただけなのか」「本人が技術を理解して作ったのか」の区別がつきません。
特にインターン志望の学生にとって、従来のポートフォリオでは技術力の証明が難しくなっています。

## 解決策
開発の**過程**（何をどう考え、どうコードを書いたか）と技術の**理解度**（クイズスコア）をセットで可視化する、新しいポートフォリオプラットフォームです。

## ターゲットユーザー
- AI支援開発で技術を習得中の学生・インターン志望者
- 彼らの技術力を正しく評価したい企業の採用担当者

## 主要機能

### 1. プロジェクト管理
- タイトル、説明文、使用技術タグ、リポジトリURL、デモURL
- ステータス管理（進行中 / 完了 / アーカイブ）
- 公開 / 非公開の切り替え

### 2. 開発ログ自動記録（MCP連携）
- Claude Code等のAI開発環境から、MCPサーバーを通じて開発過程を**自動記録**
- 記録タイプ: コード生成、デバッグ、設計判断、学習、エラー解決
- 手動入力にも対応（MCP未使用環境向け）
- source フィールドで「mcp（自動）」と「manual（手動）」を区別

### 3. 理解度チェック（LLMクイズ自動生成）
- 開発ログの内容を元に、LLM（GPT-4）が4択クイズを自動生成
- 例:「このコンポーネントでuseEffectを使った理由は？」
- 技術別のスキルスコアを算出（正答率ベース）
- 難易度3段階（easy / medium / hard）

### 4. 公開ポートフォリオ
- `/p/:username` で企業がアクセス可能
- プロジェクト一覧 + 技術別スキルスコア
- 各プロジェクトの開発ログ概要 + クイズサマリー

### 5. ダッシュボード
- 統計サマリ（プロジェクト数、ログ数、クイズ回答数、総合スコア）
- 最近のプロジェクト
- トップスキル（スコア上位の技術）

# 技術アーキテクチャ

## システム構成図

```
AI開発環境（Claude Code等）
    ↓ MCP Protocol (stdio)
MCP Server (Node.js + MCP SDK v0.5.0)
    ↓ REST API (Bearer Token)
Backend (FastAPI + Python 3.12)
    ↓ SQLAlchemy ORM
PostgreSQL（リレーショナルDB）
    +
Qdrant（ベクトルDB：埋め込み検索用）
    +
OpenAI API（GPT-4：クイズ生成 / text-embedding-3-large：埋め込み）
    ↑ REST API (JWT認証)
Frontend (React 19 + TypeScript + React Router 7)
```

## 技術スタック

| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| Frontend | React 19, TypeScript, React Router 7 | SPAとしての型安全性、最新のReact機能活用 |
| Backend | FastAPI, SQLAlchemy 2, Alembic | 非同期対応、型ヒント、自動ドキュメント生成 |
| RDB | PostgreSQL 16 | 信頼性、JSON型サポート（technologies配列等） |
| Vector DB | Qdrant | 埋め込みベクトルの高速類似検索 |
| AI/LLM | OpenAI GPT-4, text-embedding-3-large | クイズ自動生成、テキスト埋め込み |
| MCP Server | Node.js, MCP SDK | AI開発環境との標準プロトコル連携 |
| 認証 | JWT (HS256) | 短寿命60分（Web用）+ 長寿命30日（MCP用） |
| パスワード | Bcrypt | 業界標準のハッシュ化 |

## データモデル

```
User (ユーザー)
├── username, email, display_name, bio, github_url
├── plan: free / pro
│
├── Project (プロジェクト) ※複数
│   ├── title, description, technologies[], status, is_public
│   │
│   ├── DevLogEntry (開発ログ) ※複数
│   │   ├── source: "mcp" | "manual"
│   │   ├── entry_type: code_generation | debug | design_decision | learning | error_resolution
│   │   ├── summary, detail, technologies[], ai_tool
│   │   └── metadata (JSON)
│   │
│   └── QuizQuestion (クイズ問題) ※複数、LLM自動生成
│       ├── technology, question, options[4], correct_answer, explanation
│       ├── difficulty: easy | medium | hard
│       └── QuizAttempt (回答記録)
│           └── selected_answer, is_correct, time_spent_seconds
│
└── SkillScore (技術別スコア) ※技術ごとに1レコード
    └── technology, score (正答率), total_questions, correct_answers
```

## MCPサーバー設計

MCPサーバーは3つのツールを提供:

1. **record_dev_activity**: 開発ログをDBに記録（コード生成のたびに呼び出される想定）
2. **list_projects**: ユーザーのプロジェクト一覧を取得
3. **get_project_context**: プロジェクト概要 + 最近のログを取得（LLMコンテキスト用）

認証は `~/.mex/config.json` に保存した長寿命APIトークン（30日有効）を使用。

## 設計上の主な判断

1. **MCPプロトコル採用**: AI開発環境（Claude Code, Cursor等）と標準的に連携できるプロトコルを選択。開発者が意識せずにログが蓄積される仕組み。

2. **LLMによるクイズ自動生成**: 開発ログの内容からコンテキストに沿った4択問題を動的生成。汎用的な問題ではなく「その人が実際に書いたコード」に基づく問題。

3. **source フィールドによるMCP/手動区別**: MCP経由の自動記録と手動入力を同じテーブルで管理しつつ、sourceフィールドで区別。企業側は「自動記録の割合」も見られる。

4. **公開ポートフォリオ**: `/p/:username` で認証不要アクセス。GitHub Profileのように共有可能。

5. **JWT二段階トークン**: Web用（60分、セキュリティ重視）とMCP用（30日、利便性重視）を分離。

# 評価してほしい観点

以下の6つの観点から、それぞれ**良い点**と**改善すべき点**を具体的に述べてください。

## 1. ビジネス・市場性
- この課題は実際に存在するか？ターゲットユーザーの規模感は？
- 「開発過程 + 理解度」という切り口の差別化は十分か？
- 収益化（free/proプラン）の見通しは？
- 既存の競合（GitHub Profile, Zenn, Qiita等）との比較

## 2. 技術選定の妥当性
- 各レイヤーの技術選定は適切か？オーバーエンジニアリングの箇所はないか？
- MCPプロトコルの採用は時期尚早か、先見的か？
- PostgreSQL + Qdrant の組み合わせに無駄はないか？
- FastAPI + React 19 の組み合わせの長所・短所

## 3. ユーザー体験（UX）
- 学生ユーザーにとっての導入ハードル（MCP設定等）
- 企業側の閲覧体験（公開ポートフォリオの情報設計）
- 開発ログの「自動記録」がユーザーに与える心理的影響（監視感等）
- クイズ体験の自然さ・教育的価値

## 4. スケーラビリティ
- ユーザー数増加時のボトルネック予測
- LLM API（OpenAI）依存のコスト・レート制限リスク
- ベクトルDBの運用負荷

## 5. セキュリティ・プライバシー
- 開発ログに機密コードが含まれるリスクへの対処
- JWT長寿命トークン（30日）のリスク
- 公開ポートフォリオでの情報露出範囲

## 6. 総合評価と改善提案
- このプロダクトの最大の強みと最大の弱み
- MVP としての完成度（1-10）
- 優先的に改善すべきトップ3の提案
- 「これがあれば使いたい」と思う機能の提案
```
