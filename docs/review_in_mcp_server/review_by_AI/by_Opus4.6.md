## 1. ビジネス・市場性

**良い点:**
AI支援開発の普及による「技術力証明の空洞化」という課題設定は的確です。特に2025〜2026年の採用市場では、Copilot/Claude Codeの利用が当たり前になりつつあり、タイミングとして良いです。「過程＋理解度」という切り口は、GitHub ProfileやZennにはない明確な差別化軸になっています。

**改善すべき点:**
最大の懸念は**需要の両面性**です。学生側は「使うインセンティブ」がありますが、企業側が「MEXのスコアを見て採用判断する」段階に至るには、相当なブランド認知とデータの信頼性が必要です。鶏と卵の問題が発生します。free/proプランの差分が不明確で、学生が課金する動機が弱いです。むしろ企業課金モデル（採用担当が候補者を検索・閲覧するSaaS）の方が収益化の筋は良いかもしれません。

競合としてはGitHub Profile以上に、**Triplebyte（現Karat）やHackerRank、LeetCode**などのスキル証明プラットフォームとの比較が重要です。これらとの差別化を明確にすべきです。

## 2. 技術選定の妥当性

**良い点:**
FastAPI + React 19 + TypeScriptの組み合わせは、このスケールのプロダクトとして妥当です。FastAPIの型ヒントと自動ドキュメント生成はAPI開発効率が高く、MCPサーバーとの連携設計も理にかなっています。MCPプロトコルの採用は先見的と言えます。Anthropic/OpenAIがMCPを推進しており、エコシステムの成長に乗れるポジションです。

**改善すべき点:**
**Qdrant（ベクトルDB）はMVP段階ではオーバーエンジニアリング**です。クイズ生成のためだけなら、開発ログのテキストを直接LLMに渡せば十分で、埋め込み検索が必要になるのはログが大量に蓄積された後の話です。pgvectorで十分な可能性も高く、インフラの複雑性を不必要に上げています。

JWT HS256も、将来的にマイクロサービス化を見据えるならRS256の方が適切ですが、MVP段階ではHS256で問題ありません。

## 3. ユーザー体験（UX）

**良い点:**
MCP経由の自動記録というコンセプトは、開発者の手間を最小化する点で優れています。`/p/:username`の公開URLもGitHub Profileライクで直感的です。

**改善すべき点:**
**導入ハードルが最大のUXリスク**です。MCP設定、`~/.mex/config.json`の配置、トークン取得という一連の手順は、ターゲットである「AI支援開発で技術を習得中の学生」にとって高すぎる障壁になり得ます。CLIインストーラーやワンクリックセットアップが必須です。

開発ログの自動記録には**監視感・プライバシー懸念**が伴います。「何が記録されるか」「記録を事後に編集・削除できるか」を明確にし、ユーザーに制御権を持たせる設計が不可欠です。

クイズ体験については、開発直後に自分のコードについてテストされる体験が「楽しい学習」になるか「面倒な試験」になるかは、UIの作り込み次第です。ゲーミフィケーション要素（ストリーク、レベルアップ等）の導入を検討すべきです。

## 4. スケーラビリティ

**良い点:**
PostgreSQL + FastAPIの非同期処理は、中規模（数万ユーザー）までは十分対応可能です。

**改善すべき点:**
**OpenAI API依存が最大のボトルネック**です。クイズ生成はユーザーアクション起点なので制御可能ですが、コスト面ではGPT-4のクイズ生成1回あたりの単価×ユーザー数が急速に膨らみます。GPT-4o-miniへのフォールバック、キャッシュ戦略、バッチ生成などのコスト最適化が早期に必要です。text-embedding-3-largeも、smallで十分な精度が出る可能性があります。

開発ログの書き込み頻度が高い場合（MCP経由で数秒ごとに記録など）、書き込み負荷の設計も検討が必要です。

## 5. セキュリティ・プライバシー

**良い点:**
Bcryptによるパスワードハッシュ、JWT短寿命/長寿命の分離は基本を押さえています。

**改善すべき点:**
**開発ログに機密コードが混入するリスクは深刻**です。企業のプライベートリポジトリで作業中にMCPが動作した場合、秘密鍵やAPIキー、社内コードが記録される可能性があります。自動的なシークレット検出（正規表現ベースでも）と、記録前のプレビュー/承認フローが必要です。

30日トークンは、トークン漏洩時の影響が大きいです。リフレッシュトークン＋短寿命アクセストークンの組み合わせか、少なくともトークン無効化（revoke）エンドポイントが必須です。

公開ポートフォリオに開発ログの詳細がどこまで露出するかのグラニュラリティ制御（ログ単位での公開/非公開）も重要です。
