プロダクトマネージャー兼テックリードの視点から、**MEX（My Engineering Experience）**について客観的かつ厳格に評価を行います。

このプロダクトは、現在の「AIネイティブ開発時代」における採用と学習のミスマッチという、非常に鋭い課題に切り込んでいます。しかし、MVP（Minimum Viable Product）としての技術的・ビジネス的な落とし穴もいくつか見受けられます。

以下、6つの観点から詳細に評価します。

---

## 1. ビジネス・市場性

### 良い点

* **課題の解像度が高い:** 「AIが書いたコードか、本人が理解しているか判別できない」という課題は、現在進行形で採用現場の最大の悩みです。Githubの草（Contribution）がAIによる自動コミットで形骸化している今、新たな指標へのニーズは確実に存在します。
* **「プロセス＋理解度」の差別化:** 成果物（What）だけでなく、思考過程（How）と理解度（Why）をセットにするアプローチは、GitHub ProfileやQiitaにはない強力なUSP（Unique Selling Proposition）です。

### 改善すべき点・リスク

* **マネタイズの難易度（学生課金の壁）:** ターゲットが「学生」である以上、Proプラン（月額課金）のハードルは極めて高いです。就活期間は一時的であるため、LTV（顧客生涯価値）も低くなります。
* **改善案:** 学生は無料とし、企業側から「詳細レポート閲覧料」や「スカウト料」を取るB2Bモデル（リクルーティングプラットフォーム）への転換を視野に入れるべきです。


* **「不正」への対抗策:** クイズの回答自体をユーザーがChatGPTに入力して解いてしまうリスクがあります。これでは元の木阿弥です。
* **改善案:** クイズの回答制限時間（短時間化）や、ブラウザのフォーカス検知など、最低限のアンチチート対策が必要です。



## 2. 技術選定の妥当性

### 良い点

* **Backend/Frontend:** FastAPI + React 19 は現代的な標準構成であり、パフォーマンスと開発効率のバランスが良いです。
* **MCPの採用（先見性）:** Claude DesktopやCursorなど、AIエディタが標準化しつつある今、MCP（Model Context Protocol）を採用するのは非常に筋が良いです。独自プラグインを作るよりもメンテナンスコストが低く、エコシステムの波に乗れます。

### 改善すべき点・リスク

* **オーバーエンジニアリング（DB構成）:** MVP段階で PostgreSQL と Qdrant を分けるのは運用コストと複雑性を増すだけです。
* **改善案:** PostgreSQLの拡張機能 `pgvector` を使用してください。これでRDBとベクトル検索を一本化でき、トランザクション整合性も保ちやすく、インフラコストも下がります。Qdrantはスケールしてからで十分です。


* **React 19 & Router 7:** 最新すぎてライブラリの対応状況やドキュメント不足にハマるリスクがあります。学習目的も兼ねているならOKですが、最速リリースを目指すなら枯れたバージョンの方が無難です。

## 3. ユーザー体験（UX）

### 良い点

* **自動記録による負担軽減:** いちいち「今日はこれを勉強しました」と手動入力させる既存サービスは続きません。「書いているだけで勝手にログが溜まる」体験は最強のフックです。

### 改善すべき点・リスク

* **MCP導入の心理的・技術的ハードル:** 学生にとって `config.json` を編集してMCPサーバーをローカルで立てる作業は、想像以上に高いハードルです。ここで9割が離脱します。
* **改善案:** `npx create-mex-setup` のようなCLIツールを提供し、コマンド一発でセットアップが完了する仕組みが必須です。


* **クイズ体験の「ウザさ」:** 開発のたびにクイズが生成されると、ユーザーは疲弊します。
* **改善案:** リアルタイムではなく「1日の終わりにまとめて3問振り返り」のように、デイリークエスト化（ゲーミフィケーション）する方が継続率は上がります。



## 4. スケーラビリティ

### 良い点

* **非同期処理:** FastAPIの採用により、ログ受信時の高負荷には耐えやすい構造です。

### 改善すべき点・リスク

* **LLMコストの爆発:** 開発ログは大量に発生します。全てのログに対してGPT-4でクイズ生成・Embeddingを行うと、ユーザーが増えた瞬間にAPIコストが青天井になります。Freeプランユーザーが多ければ多いほど赤字になります。
* **改善案:**
1. 全ログではなく「重要なログ（エラー解決、特定規模以上のコード変更）」のみをフィルタリングしてLLMに投げる。
2. GPT-4ではなく、より安価な `gpt-4o-mini` や `Gemini Flash` への切り替え。
3. EmbeddingはローカルLLMや安価なモデルを検討する。





## 5. セキュリティ・プライバシー

### 良い点

* **トークンの分離:** Web用とMCP用で有効期限を分けている設計は適切です。

### 改善すべき点・リスク

* **機密情報の混入:** 開発ログ（特に `code_generation`）には、AWSキーやDBパスワード、環境変数が誤って含まれる可能性があります。これをそのままDBに保存し、Embeddingして企業に見せるのは致命的なリスクです。
* **改善案:** MCPサーバー側、またはバックエンド受信時に「PII（個人情報）/ シークレット除去フィルタ」を実装することが必須です。Microsoft Presidioなどの活用を検討してください。


* **公開範囲の事故:** 「非公開プロジェクト」の中にうっかり公開したくないコードが含まれていた場合、RAG（検索）のコンテキストとして漏れる可能性があります。
* **改善案:** ベクトル検索のフィルタリング条件（`is_public: true`）を厳格にテストする必要があります。



## 6. 総合評価と改善提案

### 総合スコア

**MVPとしての完成度: 7/10**
（コンセプトは9点、技術構成は7点、ビジネスモデルは5点）

### 最大の強みと弱み

* **強み:** **MCP連携による「完全自動ポートフォリオ」という体験設計。** ユーザーが何もしなくても実績が積み上がるのは強力。
* **弱み:** **「誰が金を払うのか」が不明確。** また、学生に対する導入ハードル（環境構築）とLLMコスト構造。

### 優先的に改善すべきトップ3の提案

1. **インフラの簡素化 (Use pgvector)**
* Qdrantを削除し、PostgreSQL + pgvector に一本化してください。開発・運用コストを劇的に下げられます。


2. **コスト構造の最適化 (Model Selection)**
* クイズ生成や要約には `GPT-4` は高すぎます。`GPT-4o-mini` や `Claude 3 Haiku` でも十分な精度が出ます。原価率を下げないとFreeプランは維持できません。


3. **オンボーディングの自動化 (CLI Tool)**
* 「MCPサーバーを立てる」という概念を学生に理解させる必要はありません。npmコマンド一つでセットアップが完了するCLIツールを最優先で作ってください。



### 「これがあれば使いたい」と思う機能（Killer Feature）

* **「面接対策モード」:**
* 自分の過去のコードとログを元に、**「面接官AI」が模擬面接をしてくれる機能。**
* 例：「君、このプロジェクトの `useEffect` の部分、なぜ `useLayoutEffect` にしなかったの？」と、自分のコードについてツッコミを入れてくれる。
* これがあれば、学生は「就活対策」として喜んで課金（または利用）します。これが本当の意味での「Experienceの可視化と活用」です。
