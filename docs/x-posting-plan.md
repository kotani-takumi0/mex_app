# MEX App X(Twitter) 発信戦略

> 対象: 日本語エンジニアコミュニティ
> 発信者: 個人開発者アカウント（公式アカウントではない）
> 頻度: 週3-5投稿
> 開始: 2026年2月

---

## 1. 発信戦略（全体方針）

### 1.1 投稿カテゴリと比率

| カテゴリ | 略称 | 比率 | 説明 |
|---------|------|------|------|
| 開発裏話・進捗 | DEV | 30% | 個人開発の過程、設計判断、技術選定の理由 |
| AI活用・開発Tips | AI | 25% | Claude Code, MCP, LLM活用の具体的な知見 |
| エンジニアキャリア・問題提起 | CAREER | 20% | AI時代のポートフォリオ、技術力の証明、就活 |
| プロダクト紹介・アップデート | PRODUCT | 15% | MEXの機能紹介、アップデート報告 |
| コミュニティ・雑談 | COMMUNITY | 10% | 他の個人開発者への反応、日常の気づき |

**原則**: プロダクト紹介は5投稿に1回以下。残りは「この人の視点が面白い」と思ってもらえる内容に振る。

### 1.2 ハッシュタグ戦略

**常用（1-2個を選んで使う）:**
- `#個人開発` — 個人開発者コミュニティの基本タグ
- `#AIエンジニアリング` — AI活用開発の文脈

**カテゴリ別（内容に応じて1個追加）:**
- DEV系: `#React` `#FastAPI` `#TypeScript` `#MCP` `#個人開発日記`
- AI系: `#ClaudeCode` `#AI開発` `#バイブコーディング` `#LLM`
- CAREER系: `#エンジニア転職` `#ポートフォリオ` `#就活`
- PRODUCT系: `#プロダクト開発` `#SaaS`

**ルール:**
- 1投稿あたりハッシュタグは2-3個まで（多すぎると宣伝臭が出る）
- 本文中に自然に組み込むか、末尾に1行でまとめる
- トレンドタグの無理な便乗はしない

### 1.3 投稿時間帯

| 時間帯 | 優先度 | 理由 |
|--------|--------|------|
| 7:00-8:30 | 高 | 通勤時間帯、エンジニアの朝のタイムライン確認 |
| 12:00-13:00 | 高 | 昼休み、最もエンゲージメントが高い |
| 20:00-22:00 | 中 | 夜の余暇時間、じっくり読まれやすい |
| 平日 | 高 | 週末よりエンジニア層のアクティブ率が高い |

**推奨**: 昼12時台を基本にし、朝と夜を補助的に使う。

### 1.4 エンゲージメントを高めるコツ

**構成パターン:**
1. **問いかけ型**: 「〜って思いませんか？」で始めて議論を誘発
2. **気づき共有型**: 「〜してわかったこと」で学びを共有
3. **数字型**: 具体的な数字（行数、時間、コスト）を入れる
4. **Before/After型**: 変化や改善を対比で見せる
5. **失敗談型**: うまくいかなかったことを正直に書く

**やること:**
- 他の個人開発者の投稿にリプライ・引用RTで絡む（週5回以上）
- 質問されたら必ず返信する
- スレッド（連投）は週1回まで。基本は1ツイート完結
- 画像（スクリーンショット、アーキテクチャ図）を週2回は添付

**やらないこと:**
- 「使ってください！」「登録お願いします！」の直接的なCTA
- 毎日の同じフォーマットの進捗報告（飽きられる）
- 他のプロダクトの批判
- 未実装の機能を実装済みのように書く

## 2. 2週間分の投稿テンプレート（Week 1-2）

### Week 1

---

#### Day 1（月曜日）— カテゴリ: DEV
**投稿時間**: 12:00

```
AI時代に「技術を本当に理解してるか」を証明するのが難しくなってきた。

面接官も困ってると思う。完成物だけ見ても、AIに全部任せたのか自分で考えたのかわからない。

開発の「過程」を記録するツールを個人開発で作り始めた。

#個人開発 #AIエンジニアリング
```

#### Day 2（火曜日）— 投稿なし

---

#### Day 3（水曜日）— カテゴリ: AI
**投稿時間**: 12:00

```
Claude CodeのMCPサーバー、自作してみたら開発体験が変わった。

開発中にツールを呼ぶだけで、何をどう実装したか自動で外部APIに記録できる。手動メモが不要になる。

MCPは「AIに作業させる」だけじゃなく「AIの作業を記録する」にも使える。

#ClaudeCode #MCP
```

#### Day 4（木曜日）— 投稿なし

---

#### Day 5（金曜日）— カテゴリ: CAREER

```
プログラミングスクール卒業後のポートフォリオ、最近はみんなクオリティ高い。

でも裏を返すと「差がつかない」ということ。完成物のクオリティでは差別化できない時代。

じゃあ何で差がつくかというと、「技術を自分の言葉で説明できるか」だと思う。

#ポートフォリオ #エンジニア転職
```

---

#### Day 6（土曜日）— 投稿なし

---

#### Day 7（日曜日）— カテゴリ: DEV
**投稿時間**: 20:00

```
個人開発で技術選定するとき、「使いたい技術」じゃなくて「ユーザーに最速で価値を届ける技術」で選ぶようにしてる。

今回はReact + FastAPI + PostgreSQL。新しくはないけど、情報量が多くて詰まりにくい。

個人開発は完走が最優先。

#個人開発
```

FastAPI + Stripe でフリーミアムを実装した。

決済周りは「動いたからOK」が一番危険。

#FastAPI #個人開発

#### Day 12（金曜日）— 投稿なし

---

#### Day 13（土曜日）— カテゴリ: COMMUNITY
**投稿時間**: 20:00

```
個人開発でいちばん難しいのは「完成させること」でも「集客」でもなく、「自分が本当に欲しいものを見極めること」だと思う。

作りたいものと求められるものは違う。でも自分が欲しくないものを作り続けるのは辛い。

このバランスが個人開発の面白さかも。

#個人開発
```

---

#### Day 14（日曜日）— カテゴリ: CAREER
**投稿時間**: 20:00

```
バイブコーディングで作ったアプリを面接で「自分で作りました」と言うのは嘘ではない。

ただ「どう作ったか」「なぜその技術を選んだか」「その技術の仕組みは」と聞かれたときに答えられるかが分かれ道。

AIを使うこと自体は問題じゃない。理解してるかが問題。

#バイブコーディング #エンジニア転職
```

## 4. 月間スケジュールテンプレート

投稿しない日を含めた週間リズム。

```
月: 投稿（DEV or CAREER）
火: 投稿なし — リプライ・引用RTで他の人に絡む
水: 投稿（AI or PRODUCT）
木: 投稿なし — リプライ・引用RTで他の人に絡む
金: 投稿（CAREER or DEV）
土: 任意（COMMUNITY系の軽い投稿 or 休み）
日: 任意（DEV系の振り返り or 休み）
```

週3投稿が基本、余裕があれば週4-5。無理に毎日投稿しない。

## 6. 注意事項

- 投稿前にスクリーンショットや図を添付できないか検討する（画像付きはエンゲージメント2-3倍）
- MEXのURLを貼るときはUTMパラメータを付ける（`?utm_source=x&utm_medium=social`）
- ネガティブな反応が来ても感情的に返さない。建設的な議論に持っていく
- 投稿内容に嘘や誇張を入れない。個人開発の等身大の姿を見せる
- 1ヶ月ごとに投稿の反応を振り返り、ウケたカテゴリに寄せていく
