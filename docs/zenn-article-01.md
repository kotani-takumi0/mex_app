## AIでみんな「作れる」ようになった

最近、Cursor とか Claude Code を使ってアプリを作る人がめちゃくちゃ増えた。Vibe Coding なんて言葉も出てきて、自然言語でざっくり指示するだけでそれっぽいアプリが動く時代になっている。

GitHub Copilot ユーザーが書くコードの**平均46%がAI生成**[^copilot]。Y Combinator の2025年冬バッチでは、**25%のスタートアップでコードの95%以上がLLM生成**だったらしい[^yc]。

これ自体はすごくいいことだと思う。開発のハードルが下がって、アイデアを形にしやすくなった。

でも一つ、困ったことが起きている。

[^copilot]: GitHub Copilot利用者のコード生成比率。2022年のローンチ時27%から増加。
[^yc]: [Garry Tan on X](https://x.com/garrytan/status/1897303270311489931) — W25バッチについてのポスト

## ポートフォリオが「証明」にならなくなってきた

ちょっと前まで、ポートフォリオといえば「動くもの作ってGitHubに上げる」で十分だった。Todoアプリでも、ECサイトクローンでも、「ちゃんと作れるんだな」って伝わった。

でもいま、AI使えば経験浅くてもそれなりのものが動いちゃう。

そうなると採用する側は当然こう思う。

**「これ、本当にこの人が理解して作ったの？」**

実際、レバテックの調査でも採用担当の約4割が「AIの登場でエンジニアに求めるスキルが変わった」と答えていて[^levtech]、「プログラミングスキル」の重要度は下がり、代わりに「なぜそう設計したか説明できる力」が求められるようになっている。

つまり、**「作れる」は前提になって、「分かってる」が差別化ポイントになった**。

[^levtech]: [レバテック IT人材白書 2025](https://levtech.co.jp/research/2383985/)

## じゃあどうすればいいのか

ポートフォリオに必要なものが変わったと思っている。

| 前 | いま |
|---|---|
| 完成物を見せる | 開発プロセスを見せる |
| 「動くこと」が価値 | 「理解していること」が価値 |
| GitHub のコミット履歴 | なぜそのコミットに至ったかの文脈 |

GitHub は「何をしたか」は残るけど、「なぜそうしたか」「AIの提案をどう直したか」は残らない。

かといって、開発中にいちいちメモを取るのは現実的じゃない。コード書いてるときはコードに集中したいし、あとから振り返って書こうとしても、判断の理由って驚くほど早く忘れる。

**記録は、開発しながら勝手にやってくれないと続かない。**

## それで、自分で作った

この問題をずっと考えていて、結局自分でツールを作ることにした。

**MEX** というサービスで、やってることは3つ。

### 1. 開発プロセスの自動記録

Claude Code の MCP Server として動く。開発中にファイルの変更やエラー対応が自動的に MEX に記録される。開発者は普段通りにコードを書くだけ。何も意識しなくていい。

### 2. クイズで理解度チェック

記録された内容をもとに、LLM がクイズを自動生成する。「なぜここでこのデータ構造を選んだか」「この処理の計算量は？」みたいな、コピペしただけだと答えられない問題が出る。

自分がちゃんと分かっているかの確認にもなるし、スコアとしてポートフォリオに載せられる。

### 3. 公開ポートフォリオ

開発プロセスの記録とクイズのスコアを、公開できるページにまとめられる。面接前に URL を共有するだけで、「この人はプロセスも理解度も見える化している」と伝わる。

## AI使うのは全然いい。問題は別のところ

誤解してほしくないんだけど、AIでコードを書くこと自体は全く悪くない。むしろ使わない理由がない。

問題は、**AIが書いたコードを自分が理解しているかどうかを、第三者に見せる手段がない**こと。

MEX はそこを埋めたくて作った。

まだ作りたてで足りない部分はたくさんあるけど、「プロセスを記録して、理解度を証明する」というアプローチ自体は、これからのポートフォリオに必要な方向だと思っている。

無料で使えるので、興味があったら触ってみてほしい。
